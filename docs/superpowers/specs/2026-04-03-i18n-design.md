# i18n Design — GateControl Clients

**Datum:** 2026-04-03
**Scope:** Pro Client + Community Client + client-core
**Sprachen:** Deutsch (Standard), Englisch, erweiterbar

---

## Ziel

Beide GateControl Electron-Clients (Pro + Community) mehrsprachig machen. Gemeinsame Übersetzungen leben in `@gatecontrol/client-core`, client-spezifische Strings in den jeweiligen Repos.

## Entscheidungen

- Keine externe i18n-Library — eigene schlanke Engine in client-core
- Deutsch + Englisch, vorbereitet für weitere Sprachen
- Systemsprache beim ersten Start, danach manuell umschaltbar in Settings
- Gemeinsame Basis in client-core, client-spezifische Erweiterungen pro Client
- Live-Switch ohne App-Neustart

## Architektur

### Translation-Engine (`client-core/src/i18n/index.js`)

```js
// Public API
t(key, params?)              // Übersetzt Key, optional mit Interpolation
setLocale(locale)            // Wechselt Sprache, feuert Callbacks
getLocale()                  // Gibt aktuelle Sprache zurück ('de' | 'en')
onLocaleChange(callback)     // Registriert Listener für Sprachwechsel
registerTranslations(locale, strings) // Client-spezifische Strings hinzufügen
detectLocale()               // Erkennt Systemsprache, mapped auf unterstützte
```

**Key-Format:** Verschachtelt mit Punkt-Notation — `t('status.connected')` liest `{ status: { connected: "Verbunden" } }`.

**Interpolation:** `{{ variable }}` Syntax — `t('peer.expires', { days: 3 })` → `"Dein VPN-Zugang läuft in 3 Tagen ab."`.

**Fallback-Kette:** Angefragter Key in aktueller Sprache → Key in Deutsch → Key-String selbst.

### Dateistruktur

```
gatecontrol-client-core/
  src/i18n/
    index.js              — Translation-Engine
    locales/
      de.json             — Gemeinsame deutsche Strings (~40-50 Keys)
      en.json             — Gemeinsame englische Strings

windows-client-pro/
  src/i18n/
    de.json               — Pro-spezifische Strings (RDP, erweiterte Features)
    en.json

windows-client/
  src/i18n/
    de.json               — Community-spezifische Strings (falls vorhanden)
    en.json
```

### Gemeinsame Strings (client-core)

Keys die in beiden Clients identisch vorkommen:

- `nav.*` — Navigation (Status, Einstellungen, Logs)
- `status.*` — Verbindungsstatus (Verbunden, Getrennt, Verbinde...)
- `action.*` — Buttons (Verbinden, Trennen, Speichern, Testen)
- `settings.*` — Einstellungen (Server-URL, API-Key, Kill-Switch, Split-Tunneling)
- `stats.*` — Statistiken (Endpoint, Handshake, Empfangen, Gesendet)
- `toast.*` — Benachrichtigungen (Erfolg, Fehler, Warnungen)
- `tray.*` — Tray-Menü (Fenster öffnen, Beenden)
- `update.*` — Update-Benachrichtigungen
- `peer.*` — Peer-Ablauf-Warnungen
- `dns.*` — DNS-Leak-Test
- `theme.*` — Theme-Bezeichnungen

### Client-spezifische Strings

**Pro Client:**
- `rdp.*` — RDP-Karten, Verbindungsstatus, Fehlermeldungen
- `services.*` — Erreichbare Dienste (erweiterte Darstellung)

**Community Client:**
- Voraussichtlich keine eigenen Strings nötig (alles in core abgedeckt)

## Renderer-Integration

### HTML: `data-i18n`-Attribute

```html
<!-- Textinhalt -->
<span data-i18n="nav.status">Status</span>
<button data-i18n="action.connect">Verbinden</button>

<!-- Attribute -->
<input data-i18n-placeholder="settings.serverUrl" placeholder="Server-URL">
<button data-i18n-title="action.testTooltip" title="Verbindung testen">
```

Deutsche Fallback-Texte bleiben als Inhalt stehen — werden beim `updateDOM()` überschrieben.

### JS: `updateDOM()` Funktion

```js
function updateDOM() {
  document.querySelectorAll('[data-i18n]').forEach(el => {
    el.textContent = t(el.dataset.i18n);
  });
  document.querySelectorAll('[data-i18n-placeholder]').forEach(el => {
    el.placeholder = t(el.dataset.i18nPlaceholder);
  });
  document.querySelectorAll('[data-i18n-title]').forEach(el => {
    el.title = t(el.dataset.i18nTitle);
  });
  document.documentElement.lang = getLocale();
}
```

Wird aufgerufen bei:
1. App-Start (nach Locale-Erkennung)
2. `onLocaleChange` Callback

### Dynamische Strings

Strings die im JS generiert werden (Toasts, Status-Updates, dynamische Elemente) nutzen direkt `t()`:

```js
// Vorher
showToast('Verbindung getrennt', 'error');
// Nachher
showToast(t('toast.disconnected'), 'error');
```

## Main-Prozess-Integration

### Tray-Menü + Notifications

Main-Prozess braucht eigenen Zugriff auf Translations (kein DOM):

```js
const { t, setLocale } = require('@gatecontrol/client-core/src/i18n');
// + client-spezifische Strings registrieren
```

### IPC für Sprachwechsel

```
Renderer → 'locale:set' (locale) → Main
Main → Tray neu bauen, Locale speichern
Main → 'locale:changed' (locale) → Renderer
Renderer → updateDOM()
```

### preload.js Erweiterung

```js
locale: {
  set: (locale) => ipcRenderer.invoke('locale:set', locale),
  get: () => ipcRenderer.invoke('locale:get'),
  onChange: (callback) => ipcRenderer.on('locale:changed', (_, locale) => callback(locale))
}
```

## Spracherkennung

1. **Config prüfen** — gespeicherte Sprache vorhanden? → verwenden
2. **Systemsprache** — `app.getLocale()` im Main-Prozess (Electron API)
3. **Mapping:** `de`, `de-DE`, `de-AT`, `de-CH` → `de` | alles andere → `en`
4. **Persistenz:** `config:set('locale', 'en')` über bestehenden Config-Mechanismus

## Settings-UI

Neuer Eintrag im Settings-Bereich:

```
Sprache / Language
[🇩🇪 Deutsch     ▼]
```

Dropdown mit Flagge + Sprachname in der jeweiligen Sprache. Bei Auswahl: sofortiger Live-Switch.

## Reihenfolge

1. Translation-Engine in client-core implementieren + gemeinsame Locale-Dateien
2. Pro Client migrieren (größerer Client = mehr Edge Cases)
3. Community Client migrieren (gleiche Patterns wiederverwenden)
